  <files></files>
  <action>
Deploy the new conf.d fragments and remove old config.toml symlinks. Follow the established Phase 4/5 deployment pattern.

**Step 1: Remove old symlinks that stow needs to replace.**
Check if `~/.config/mise/config.toml` exists as a symlink (from the old merged config). If so, remove it. Also check `~/.tool-versions` for the stale symlink.

```bash
# Remove old config.toml symlink if it exists
[ -L ~/.config/mise/config.toml ] && rm ~/.config/mise/config.toml
# Remove stale .tool-versions symlink if it exists
[ -L ~/.tool-versions ] && rm ~/.tool-versions
```

**Step 2: Restow dev-tools and node modules.**
Use `stow --no-folding` to ensure individual file symlinks (not directory symlinks) are created. Since the old config.toml source files were deleted in Plan 01, stow will not create a symlink for them.

```bash
cd /path/to/dotfiles/modules
# Restow dev-tools (creates conf.d/dev-tools.toml symlink, updates shell activation scripts)
stow --no-folding --restow --target="$HOME" dev-tools
# Restow node (creates conf.d/node.toml symlink)
stow --no-folding --restow --target="$HOME" node
```

Note: The `mergeable_files` entries in config.yml reference files that no longer exist in the module trees (deleted in Plan 01), so the merge process becomes a no-op. The config.yml cleanup is Phase 7 scope.

**Step 3: Verify symlinks exist.**
```bash
ls -la ~/.config/mise/conf.d/
# Expected: dev-tools.toml -> stow symlink, node.toml -> stow symlink
```

If stow reports conflicts (e.g., existing non-stow-managed files in conf.d/), remove the conflicting files first, then restow. This matches the pattern from Phase 4 and 5 deployments.

  </verify>
