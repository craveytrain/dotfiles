  <action>
Read the full CODING_STANDARDS.md first. Add a new major section titled "## conf.d Convention" (or "## Shell Configuration Conventions") covering:

1. **Overview** - 2-3 sentences: Modules contribute configuration fragments to shell-specific conf.d directories. These are stowed into place and sourced at shell startup, so edits are live on git pull without Ansible redeploy.

2. **Prefix Range Table:**

| Range | Purpose | Current Modules |
|-------|---------|----------------|
| 00-19 | Core shell config (shell-specific foundations) | zsh (10), fish (10) |
| 20-49 | Reserved for future core modules | (unused) |
| 50-69 | Module features (utilities, editor, aliases) | editor (50), shell (50) |
| 70-79 | Reserved for future feature modules | (unused) |
| 80-99 | Late-loading (runtime managers, tools needing PATH) | dev-tools (80) |

Note: Lower numbers load first. Runtime managers like mise go high (80+) because they may depend on PATH being set.

3. **Fragment Naming Pattern:**
- Zsh: `NN-module-description.sh` in `files/.zsh/conf.d/`
- Fish: `NN-module-description.fish` in `files/.config/fish/conf.d/`
- Mise: `module-name.toml` in `files/.config/mise/conf.d/` (no numeric prefix, mise has no ordering concerns)

4. **Fragment Header Format** with examples for each type:
- Zsh: line 1 `# shellcheck shell=zsh`, line 2 `# {module} module - {description}`, blank line, content
- Fish: line 1 `# {module} module - {description}`, blank line, content
- Mise: line 1 `# {module} module - {description}`, line 2 `# Managed by dotfiles {module} module`, blank line, TOML content with own `[tools]`/`[settings]` headers

5. **End-to-End Example** - Walk through adding a hypothetical "python" module that:
   - Creates `files/.zsh/conf.d/50-python-env.sh` with a PYENV_ROOT export
   - Creates `files/.config/fish/conf.d/50-python-env.fish` with the fish equivalent
   - Creates `files/.config/mise/conf.d/python.toml` with `[tools]` section listing python version
   - Adds `stow_dirs: [python]` to config.yml
   - Shows the stow deployment placing symlinks

Keep the existing code style. Use the same markdown heading levels and formatting patterns as the rest of the document. If the document uses `###` for subsections, follow that.

Also remove any existing references to mergeable_files or merge behavior if present in CODING_STANDARDS.md.
  </action>
